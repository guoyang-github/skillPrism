# 第 1 章：为什么要评估 Skill

> 学习目标：理解 Skill 质量问题的根源，以及 skillPrism 的核心设计哲学。

## 1.1 一个常见场景

你正在维护一个 Agent 的 skills 目录。里面有几十个 Skill：

- 有的 `SKILL.md` 写得很长，但关键参数没写清楚；
- 有的示例代码复制过来就能跑，有的却缺少依赖；
- 有的 Skill 文档看起来不错，但一跑 benchmark 就挂；
- 有的修改后 Rubric 分数提高了，实际能力却下降了。

如果没有一个统一的测量框架，优化就全靠主观判断，很容易陷入「改一点、崩一点」的循环。

## 1.2 skillPrism 要回答的问题

skillPrism 把 Skill 质量拆成可客观测量的维度：

| 问题 | skillPrism 的回应 |
|---|---|
| 这个 Skill 文档是否完整？ | D1（目录与元数据）、D2（文档可理解性） |
| 示例能不能跑通？ | D3（可执行性/正确性）、smoke test |
| 依赖是否可复现？ | D4（环境/依赖可复现） |
| 领域知识是否准确？ | D5（领域准确性） |
| LLM 能不能正确调用？ | D6（LLM 可调用性） |
| 是否稳定、可维护？ | D7、D8 |
| 有没有安全风险？ | D9（安全与可信） |

## 1.3 核心设计哲学

skillPrism 坚持三层分离：

```text
Skill 入口层（自然语言）
   ↓
可选能力层（editor / LLM judge，由用户接入）
   ↓
引擎层（skillPrism，只做客观测量）
```

**引擎不调用 LLM**。评分、benchmark、回归判断、回滚都是确定性的。这样 Agent 可以把 skillPrism 当作一个可复用的测量工具，而不必担心 provider 依赖或成本。

## 1.4 本章小结

- Skill 质量需要可重复、可对比的测量。
- skillPrism 用 9 维度 Rubric + Benchmark + 优化循环来提供这套测量。
- 引擎层保持无 LLM，可选能力层交给用户选择。

2. 对照 D1–D9，初步判断它们分别属于哪个维度。
