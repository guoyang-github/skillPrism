# 新手操作手册：用说话的方式驱动 skillPrism

> 这份手册写给**第一次接触 skillPrism、不想记命令行**的你。
> 从头到尾，你只需要做一件事：**像对同事说话一样，告诉 Agent 你想干什么**。
> Agent 会替你调用 skillPrism 的所有功能，你负责看结果、做决定。

---

## 一、三分钟搞懂 skillPrism 是干什么的

你写了一些"技能"（Skill，就是教 AI 干活的说明书，通常是一个叫 `SKILL.md` 的文件）。你不知道它们写得好不好、AI 照着做能不能做对、该怎么改进。

skillPrism 就是来解决这三个问题的：

| 你的疑问 | skillPrism 的回答方式 |
|---|---|
| 这份说明书**写得好不好**？ | 给说明书打分（满分 100），并告诉你哪里扣了分 |
| AI 照着做，**做得对不对**？ | 出考题（Benchmark），让 AI 真做一遍，对照标准答案评分 |
| **怎么改**才能更好？ | 找出最弱的地方，给出修改建议，改完再考一次，进步了才保留 |

你可以把 skillPrism 想象成一位**质检员 + 教练**：体检（打分）、考试（做题）、特训（改进），一条龙。

### 谁干活，谁动脑

这是最重要的一条原理，理解了它，后面全都顺理成章：

- **skillPrism 引擎**：负责**测量**。打分、判卷、对比、回滚——全部是确定的、可重复的机械工作，它自己不用 AI，也不会瞎猜。
- **Agent（你的 AI 助手，如 Claude Code）**：负责**动脑**。读懂你的自然语言、编辑技能说明书、回答主观问题。

所以你全程只需要和 Agent 对话。Agent 在背后替你调用引擎，再把结果翻译成人话讲给你听。

### 三条安全底线（先吃个定心丸）

1. **默认不改任何东西**。所有可能修改文件的动作，默认只是"演习"，把打算怎么做展示给你看。
2. **你点头才算数**。Agent 每次要改技能说明书之前，必须先说明计划、展示改动内容，等你确认。
3. **改坏了能回来**。每次改进都会先记录"改动前的成绩"，改完复考，没进步就自动恢复原样。

## 二、安装与配置（一次性，约 10 分钟）

整个过程分四步。具体的命令在 [安装与环境](install.md) 里，这里只说**做什么、为什么**；如果你连命令行都不想碰，可以把安装文档丢给 Agent，说"按这个文档帮我装好"。

**第 1 步：安装 skillPrism 本体。**
它是一个 Python 软件包，用 pip 安装（需要 Python 3.9 或更新版本）。装好之后，你的电脑里就多了一套"质检工具"。

**第 2 步：按需安装可选组件。**
如果你要测试数据分析类技能（比如单细胞测序分析），还需要额外的科学计算库；如果暂时用不上，可以跳过，以后再补。

**第 3 步（关键）：把"翻译官"交给 Agent。**
skillPrism 自带一个叫 `skill-prism` 的技能，它的作用是**教 Agent 听懂你的话**——你一说"打个分"，Agent 就知道该调用哪个工具、带什么参数。
把它复制到 Agent 能自动读取的技能目录里（比如 Claude Code 的 `.claude/skills/` 目录），以后每个会话都生效，不用重复设置。

**第 4 步：验证安装。**
对 Agent 说一句："帮我检查一下 skillPrism 装好了没有。"Agent 会运行一个示例技能，告诉你一切正常。

!!! tip "配置就这些"
    没有账号、没有密钥、没有服务器。skillPrism 完全在你的电脑上运行，不会把数据发到任何地方。

## 三、你的项目里会有哪些文件夹

用 skillPrism 管理技能，你的项目目录大致长这样。先混个眼熟，后面每一步都会提到它们：

```text
你的项目/
├── skills/                # 你的技能们，每个技能一个文件夹
│   └── my-skill/
│       └── SKILL.md       # 技能说明书（被打分、被改进的就是它）
├── benchmarks/            # 考题库：每个技能一套考题
│   └── my-skill/
│       ├── data/          # 考题用的输入材料
│       ├── expected/      # 标准答案
│       └── registry.yaml  # 考题登记表（考什么、怎么判分）
├── reports/               # 体检报告、质量报告的存放处
├── artifacts/             # 过程记录：每次考试、每次改进的历史
└── .claude/skills/        # 给 Agent 看的技能入口（翻译官住这里）
```

记住三个要点就够了：

- **`skills/` 是你精心维护的"资产"**——skillPrism 默认绝不往里写任何自动生成的文件。
- **`benchmarks/` 是考试专用区**——题目、答案、判分规则都在这里，和技能说明书分开。
- **`reports/` 和 `artifacts/` 是"输出区"**——所有自动生成的报告、历史记录都往这里放，不会污染你的技能文件夹。

## 四、评分原则：分数是怎么来的

### 九个维度，各管一摊

skillPrism 从九个角度给技能说明书打分，每个角度有固定权重（越重要的占比越高）：

| 维度 | 白话解释 | 权重 |
|---|---|---|
| D1 结构规范 | 文件齐不齐、格式对不对、元数据写没写 | 10% |
| D2 讲得清不清楚 | 说明书写得明不明白，有没有例子、有没有常见错误提示 | 15% |
| D3 能不能执行 | 给的代码示例语法对不对、能不能跑 | 18% |
| D4 环境可复现 | 依赖写清楚没有，换个电脑还能不能装出来 | 12% |
| D5 领域准确性 | 专业内容靠不靠谱（比如生物信息的概念用对没有） | 15% |
| D6 AI 可调用性 | AI 容不容易读懂并照做，有没有模棱两可的话 | 10% |
| D7 性能与稳健 | 有没有考虑超时、大数据量、异常情况 | 8% |
| D8 可维护性 | 结构清不清晰，以后好不好改 | 4% |
| D9 安全可信 | 有没有危险操作（删库、强推）且不加警告 | 8% |

每个维度打 1 到 5 分，按权重加权后换算成 100 分制，再折算成等级（A/B/C/D）。

### 机器检查 + AI 评委 + 出题实测，一次体检全做完

默认的全面评估由三块拼成，各管一段：

- **机器检查**：数标题、查关键词、验证语法、扫描危险操作。快、免费、结果稳定，但只能看"表面功夫"——说明书可以靠堆关键词骗分。
- **AI 评委**：Agent 自己当第二位评委，复核"讲得清不清楚""专业不专业"这类机器只能粗判的主观维度。通过 Agent 操作时**不需要任何额外配置**——Agent 本身就是大模型。
- **出题实测**：机器检查和评委看的都是说明书，实测看的是**真用起来行不行**——带着技能和不带技能各做一遍同一批任务，对比出技能的真实贡献。

三块互相补漏：机器检查防"不规范"，评委防"骗分"，实测防"高分低能"。赶时间可以说"快速打个分"，只跑机器检查。

### 打分 ≠ 考试

这是新手最容易混淆的地方：

- **打分（评估）**看的是**说明书本身**——就像评价一份菜谱写得清不清楚。
- **考试（Benchmark）**看的是**做出来的菜**——让 AI 真做一遍，和标准答案对比。

一份菜谱写得天花乱坠，炒出来的菜未必能吃。所以**两个都要做**，先打分、后考试，是标准流程。

## 五、全流程操作：跟着做一遍

下面是最常用的八个动作，按真实使用顺序排列。每一步都告诉你：**你说什么**、**背后发生什么**、**你得到什么**、**需要你做什么**。

### 第 1 步：体检——"全面评估一下这个技能"

> **你说**："全面评估一下 my-skill。"

**背后发生什么**：一次体检做三件事，最后合成一份成绩单：

1. **机器检查**：引擎读完整个技能文件夹，按九个维度逐项检查，给出每一维的得分和扣分原因；
2. **AI 评委复核**：Agent 自己当第二位评委，对"讲得清不清楚""专业不专业"这类主观维度独立打分（默认两位评委取中位数），与机器分混合；
3. **出题实测**：Agent 从技能的真实使用场景里拟出 2–3 条测试 prompt，让"带着技能"和"不带技能"的两个独立 Agent 各做一遍，再请第三个 Agent 当裁判对比——技能到底有没有用、带不带差多少，一目了然。实测结果会直接反映到总分里。

**你得到什么**：一份三合一成绩单——总分、等级、每个维度的得分和扣分点，外加评委意见和"带/不带技能"的实测对比。

**需要你做什么**：确认一件事——Agent 拟好测试 prompt 后会先拿给你看，觉得不像真实用户会说的话就改两句。其余全自动。整个体检要调用大模型、跑实测，大约几分钟（不是几秒），也会消耗一些大模型额度。

!!! tip "赶时间？有快速模式"
    说"快速打个分"，Agent 就只做机器检查，几秒钟出结果，适合日常小改后的随手一查。但**第一次评估、发布前、优化前后对比，都建议用完整的全面评估**——只有实测能告诉你技能是不是真的有用。

### 第 2 步：诊断——"哪里最弱？为什么？"

> **你说**："这个技能哪里扣分最多？详细说说。"

**背后发生什么**：Agent 用详细模式重新评估，把每个扣分点的证据翻给你看——是缺文件、是措辞模糊、还是示例代码有语法错误；AI 评委对主观维度的意见和实测对比结果也会一并展开。

**你得到什么**：一张按严重程度排序的"问题清单"，通常附改进建议。

**需要你做什么**：判断哪些问题值得修。不是每个扣分点都要修——比如"性能稳健性"对一个一次性脚本可能无所谓。

### 第 3 步：出题——"帮我建一套考题"

> **你说**："帮我给 my-skill 建一个 benchmark。"

**背后发生什么**：这是全过程中**最需要你参与**的一步，Agent 会逐项和你确认：

1. **考什么任务**：这个技能的核心能力是什么？（比如"对单细胞数据做细胞类型注释"）
2. **用什么数据**：需要一份真实或模拟的输入数据，你提供，或让 Agent 帮你生成模拟数据。
3. **标准答案是什么**：正确答案长什么样？你提供，或 Agent 用可靠方法算出来、你审核。
4. **怎么判分**：哪些指标算通过？（比如"注释结果和标准答案的一致度超过 50%"）

确认完毕后，Agent 把考题登记到 `benchmarks/my-skill/` 下。

**你得到什么**：一套可以反复使用的考题。以后每次技能改动，都能用同一套题复考。

**需要你做什么**：提供或审核数据和标准答案。**标准答案的质量直接决定考试的意义**——这一步值得花时间。

!!! note "为什么考题要单独建"
    打分（第 1 步）任何人都能立刻做，但考试需要"标准答案"，而标准答案只有懂业务的人才能给出。这就是为什么建考题是单独一步，而且需要你的专业判断。

### 第 4 步：考试——"测一下这个技能能不能做对"

> **你说**："测一下 my-skill。"

**背后发生什么**：Agent 拿出考题，让被测技能真做一遍（或由 Agent 扮演技能执行一遍），把产出和标准答案对比，按判分规则给出"通过/不通过"和各项指标得分。

根据你的情况，考试有三种考法，Agent 会帮你选：

| 考法 | 适合什么时候 |
|---|---|
| 直接判卷 | AI 已经做完了一遍，你只想验证结果对不对（默认方式） |
| 现场做题 | 你有一段技能生成的代码，让引擎当场运行、当场判分 |
| 由易到难分级考 | 先跑 10 秒的小题，过了再跑几分钟的中题，最后才上真实大数据——省时间、省算力 |

**你得到什么**：每个考题的通过状态和指标数值，哪道题没过、差在哪。

**需要你做什么**：通常不用干预。如果考题需要真实数据或 GPU，Agent 会提前问你。

### 第 5 步：特训——"帮我优化这个技能"

> **你说**："帮我优化 my-skill。"

**背后发生什么**：这是 skillPrism 最核心的闭环，Agent 会一轮一轮地带你走：

1. **记录当前成绩**（叫"基线"），作为进步的参照；
2. **指出最弱维度**，并给出修改计划——**此时会停下来等你批准**；
3. 你批准后，Agent 修改技能说明书（默认**只改说明书**，不动代码和数据）；
4. **把改动内容完整展示给你**——加了什么、删了什么，一目了然；
5. 你确认后，引擎重新打分：**进步了才保留，没进步自动恢复原样**；
6. 进入下一轮，直到没有明显提升或你喊停。

**你得到什么**：一个分数更高、质量更好的技能说明书，以及每一轮的改动记录。

**需要你做什么**：两轮确认——**改之前**批准计划，**改之后**确认保留。这是整个流程里你最重要的控制权，不要图省事全部跳过。

!!! warning "两个不要"
    - 不要让 Agent 一轮里改好几个维度——一次只改一个问题，才知道是什么起了作用。
    - 不要在没有考题（第 3 步）的情况下盲目优化——只看打分优化，容易把说明书改成"应试作文"：分数涨了，实际效果没变。打分和考试一起进步，才是真进步。

### 第 6 步：复测——"确认一下改完没有退步"

> **你说**："再考一遍，确认优化没有副作用。"

**背后发生什么**：Agent 重新跑一遍评估和考题，和优化前的基线对比：分数是否保持、考题是否全部仍然通过。如果发现"说明书分涨了但考题反而退步"，会提示你这一轮优化有问题。

**你得到什么**：优化前后的对比结论。

**需要你做什么**：如果退步了，决定回滚到哪一版（Agent 保留了每一轮的记录）。

### 第 7 步：出报告——"给我一份质量报告"

> **你说**："给所有技能做个体检，出一份总报告。"

**背后发生什么**：Agent 批量评估项目里的所有技能，按分数排名，生成一份总报告，同时把这次成绩记入历史档案。

**你得到什么**：`reports/` 下的质量报告——谁最好、谁最差、整体趋势是涨是跌。定期跑一次，就能看到每个技能的质量曲线。

**需要你做什么**：什么都不用。报告是自动归档的。

### 第 8 步（可选）：接入自动化——"以后每次改代码都自动检查"

> **你说**："把质量检查接入 CI。"

**背后发生什么**：Agent 帮你在代码仓库里配置自动检查：每次提交代码，自动给技能打分、跑考题，**分数低于历史最高就报警**（这叫"棘轮"——像棘轮扳手一样只能前进不能后退），防止技能质量在团队协作中悄悄退化。

**你得到什么**：一套无人值守的质量门禁。

**需要你做什么**：首次配置后就不用管了，出问题它会自动报警。

## 六、常见问题

**我还没有任何技能，能用什么？**
先让 Agent 帮你写一个最简单的技能（哪怕只是"读 CSV 出统计摘要"），然后从第 1 步开始走一遍流程。skillPrism 自带模板可以参考。

**我没有真实数据，能建考题吗？**
可以。让 Agent 用固定随机种子生成模拟数据——模拟数据的好处是可重复、体积小、跑得快。等业务成熟后再补真实数据考题。

**分数低就说明技能差吗？**
不一定。机器检查看的是"表面规范"，一份内容很好但格式不规范的说明书也会低分。看扣分点比看总分更重要：如果扣分都在"缺文件、缺示例"这类地方，补起来很快；如果扣分在"领域准确性"，才需要认真对待。

**一定要配置大模型（LLM）吗？**
不用额外配置。你本来就是通过 Agent 操作的，Agent 自己就是大模型——AI 评委、出题实测、自动改说明书都由它完成，不需要密钥或外部服务。只有抛开 Agent、只用纯命令行跑引擎时，才退化为只做机器检查。

**我有很多个技能，怎么管理？**
每个技能独立打分、独立建考题。第 7 步的批量报告就是给多技能项目准备的——一眼看出谁最需要优化。

**Agent 改坏了我的技能怎么办？**
改不回来才算事故。每一轮优化前都记录了基线，引擎默认"没进步就自动回滚"；你也可以随时让 Agent 展示改动历史、回滚到任意一轮。再加上你每轮都有确认权，三重保险。
